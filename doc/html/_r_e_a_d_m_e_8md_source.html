12 # MoonGen Packet Generator
13 
14 MoonGen is a high-speed scriptable packet generator.
15 The whole load generator is controlled by a Lua script: all packets that are sent are crafted by a user-provided script.
16 Thanks to the incredibly fast LuaJIT VM and the packet processing library DPDK, it can saturate a 10 GBit Ethernet link with 64 Byte packets while using only a single CPU core.
17 MoonGen can achieve this rate even if each packet is modified by a Lua script. It does not rely on tricks like replaying the same buffer.
18 
19 MoonGen can also receive packets, e.g. to check which packets are dropped by a
20 system under test. As the reception is also fully under control of the user's
21 Lua script, it can be used to implement advanced test scripts. E.g. one can use
22 two instances of MoonGen that establish a connection with each other. This
23 setup can be used to benchmark middle-boxes like firewalls.
24 
25 Reading the example script [quality-of-service-test.lua](https://github.com/emmericp/MoonGen/blob/master/examples/quality-of-service-test.lua) is a good way to learn more about our scripting API as this script uses most features of MoonGen.
26 
27 MoonGen focuses on four main points:
28 
29 * High performance and multi-core scaling: > 15 million packets per second per CPU core
30 * Flexibility: Each packet is crafted in real time by a user-provided Lua script
31 * Precise and accurate timestamping: Timestamping with sub-microsecond precision on commodity hardware
32 * Precise and accurate rate control: Reliable generation of arbitrary traffic patterns on commodity hardware
33 
34 You can have a look at [our slides from a recent talk](https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/Slides.pdf) or read a draft of [our paper](http://arxiv.org/ftp/arxiv/papers/1410/1410.3322.pdf) [1] for a more detailed discussion of MoonGen's internals.

37 # Architecture
38 
39 MoonGen is basically a Lua wrapper around DPDK with utility functions for packet generation.
40 Users write custom scripts for their experiments. It is recommended to make use of hard-coded setup-specific constants in your scripts. The script is the configuration, it is beside the point to write a complicated configuration interface for a script.
41 
42 The following diagram shows the architecture and how multi-core support is handled.
43 
44 <p align="center">
45 <img alt="Architecture" src="https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/img/moongen-architecture.png" srcset="https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/img/moongen-architecture.png 1x, https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/img/moongen-architecture@2x.png 2x"/>
46 </p>
47 
48 Execution begins in the *master task* that must be defined in the user's script.
49 This task configures queues and filters on the used NICs and then starts one or more *slave tasks*.
50 
51 Note that Lua does not have any native support for multi threading.
52 MoonGen therefore starts a new and completely independent LuaJIT VM for each thread.
53 The new VMs receive serialized arguments: the function to execute and arguments like the queue to send packets from.
54 Threads only share state through the underlying library.
55 
56 The example script [quality-of-service-test.lua](https://github.com/emmericp/MoonGen/blob/master/examples/quality-of-service-test.lua) shows how this threading model can be used to implement a typical load generation task.
57 It implements a QoS test by sending two different types of packets and measures their throughput and latency. It does so by starting two packet generation tasks: one for the background traffic and one for the prioritized traffic.
58 A third task is used to categorize and count the incoming packets.

61 # Hardware Timestamping
62 Intel commodity NICs like the 82599, X540, and 82580 support time stamping in hardware for both transmitted and received packets.
63 The NICs implement this to support the IEEE 1588 PTP protocol, but this feature can be used to timestamp almost arbitrary UDP packets.
64 The NICs achieve sub-microsecond precision and accuracy.
65 
66 A more detailed evaluation can be found in [our paper](http://arxiv.org/ftp/arxiv/papers/1410/1410.3322.pdf) [1].

68 # Rate Control
69 Precise control of inter-packet gaps is an important feature for reproducible tests.
70 Bad rate control, e.g. generation of undesired micro-bursts, can affect the behavior of a device under test [1].
71 However, software packet generators are usually bad at controlling the inter-packet gaps [2].
72 
73 The following diagram illustrates how a typical software packet generator tries to control the packet rate.
74 
75 <p align="center">
76 <img alt="Software Rate Control" src="https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/img/ratecontrol-traditional-software.png" srcset="https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/img/ratecontrol-traditional-software.png 1x, https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/img/ratecontrol-traditional-software.png@2x 2x"/>
77 </p>
78 
79 It simply tries to wait for a specified time after sending a packet.
80 Network APIs often abstract NICs in a way that indicates that the API pushes a packet to the NIC, so this technique might seem reasonable.
81 However, NICs do not work that way. Sending a packet to the API merely places it in a queue in the main memory.
82 It is now up to the NIC (which may or may not be notified by the API about the new packet immediately) to fetch and transmit the packet asynchronously at a convenient time.
83 
84 This means that trying to push packets to a NIC is futile.
85 This is especially important at rates above 1 GBit/s where nanosecond-level precision is required (length of a minimal sized packet at 10 GBit/s: 67.2 nanoseconds).
86 Sending a single packet requires at least two round trips across the PCIe bus: One to notify the NIC about the updated queue, one for the NIC to fetch the packet. Each PCIe operation introduces latencies and jitter in the nanosecond-range.
87 
88 Another problem with this approach is that the queues, and therefore batch processing, cannot be used.
89 However, batch processing is an important technique to achieve line rate at high packet rates [3].
90 
91 MoonGen therefore implements two ways to prevent this problem.
92 
93 ## Hardware Rate Control
94 
95 Intel 10 GbE NICs (82599 and X540) support rate control in hardware.
96 This can be used to generate CBR or bursty traffic with precise inter-departure times.
97 
98 [Our paper](http://arxiv.org/ftp/arxiv/papers/1410/1410.3322.pdf) [1] features a detailed evaluation of this feature and compares it to software methods.
99 
100 ## Better Software Rate Control
101 The hardware supports only CBR traffic. Other traffic patterns, especially a Poisson distribution, are desirable.
102 
103 The problem that software rate control faces is that it needs to generate an 'empty space' on the wire.
104 We circumvent this problem by sending bad packets in the space between packets instead of trying to send nothing.
105 The following diagram illustrates this concept.
106 
107 <p align="center">
108 <img alt="Better Software Rate Control" src="https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/img/ratecontrol-moongen-software.png" srcset="https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/img/ratecontrol-moongen-software.png 1x, https://raw.githubusercontent.com/emmericp/MoonGen/master/doc/img/ratecontrol-moongen-software@2x.png 2x"/>
109 </p>
110 
111 A bad packet is a packet that is not accepted by the DuT (device under test) and filtered in hardware before it reaches the software. These packets are shaded in the figure above.
112 We currently use packets with an invalid CRC and an invalid length if necessary.
113 All common NICs drop such packets immediately in hardware as further processing of a corrupted packet is pointless.
114 This does not affect the running software.
115 [Our paper](http://arxiv.org/ftp/arxiv/papers/1410/1410.3322.pdf) contains a measurement which shows that this is the case.
116 
117 If the DuT's NIC does not do this or if a hardware device is to be tested, then a switch can be used to remove these packets from the stream to generate 'real' space on the wire.
118 The effects of the switch on the packet spacing needs to be analyzed carefully, e.g. with MoonGen's inter-arrival.lua example script.

128 # Installation
129 
130 1. Install the dependencies (see below)
131 2. git submodule update --init
132 2. ./build.sh
133 3. ./setup-hugetlbfs.sh
134 4. Run MoonGen from the build directory
135 
136 Note: You can also use the script `bind-interfaces.sh` to bind all currently unused NICs (no routing table entry in the system) to DPDK/MoonGen. `build.sh` calls this script automatically.
137 Use `deps/dpdk/tools/dpdk_nic_bind.py` to unbind NICs from the DPDK driver.

156 # Frequently Asked Questions
157 
158 ### Which NICs do you support?
159 Basic functionality is available on all [NICs supported by DPDK](http://dpdk.org/doc/nics).
160 Hardware timestamping is currently supported and tested on Intel 82599, X540 and 82580 chips.
161 Hardware rate control is supported and tested on Intel 82599 and X540 chips.
162 
163 
164 ### How is MoonGen different from SnabbSwitch?
165 [SnabbSwitch](https://github.com/SnabbCo/snabbswitch) is a framework for packet-processing in Lua.
166 There are a few important differences:
167 
168 * MoonGen comes with explicit multi-core support in its API, SnabbSwitch does not
169 * MoonGen focuses on efficient packet generation, SnabbSwitch is a more generic framework
170 * Our API is designed for packet-generation tasks, i.e. writing a packet generator script for MoonGen is a lot easier than writing one for SnabbSwitch
171 * We implement driver-like functionality for hardware functions required by packet generators: timestamping, rate control, and packet filtering
172 * SnabbSwitch reimplements the NIC driver in Lua, we rely on the DPDK driver for most parts
173 
174 
175 ### Why does MoonGen use DPDK instead of SnabbSwitch as driver?
176 We decided for DPDK as back end for the following reasons:
177 
178 * DPDK is faster for raw packet IO. This is not really a drawback for the use cases SnabbSwitch is designed for where IO is only a small part of the processing. However, for packet generation, especially with small packets, the share of packet IO is significant and the performance of SnabbSwitch is not sufficient here.
179 * DPDK provides a stable and mature code base whereas SnabbSwitch is a relatively young project.
180 * DPDK currently supports more NICs (with stable and mature drivers) than SnabbSwitch. (We do not want to write our own drivers or debug existing ones.)
181 * Lack of multi-core support. (Only possible by starting SnabbSwitch more than once.)
182 
183 Note that this might change. Using DPDK also comes with disadvantages like its bloated build system and configuration.
